## 1. Общая концепция и дизайн-система

### 1.1 Характер интерфейса

Админ-панель — внутренний инструмент для команды платформы. Пользователи — несколько операторов, которые работают с ней ежедневно. Приоритеты:

- **Информационная плотность.** Администраторы должны видеть максимум данных без лишних переходов.
- **Чёткость действий.** Деструктивные операции (досрочное закрытие, деактивация) должны быть защищены дополнительным подтверждением.
- **Скорость.** Таблицы, фильтры, быстрый поиск — интерфейс работника, а не витрина.

### 1.2 Визуальный стиль

**Направление:** Строгий «editorial utility» — плоские формы, монохромная основа с одним акцентным цветом, минимум теней, максимум читаемости. Вдохновение: Linear, Vercel Dashboard, Ramp.

**Цветовая палитра (CSS переменные):**

| Слой | Решение |

### 2.3 Axios interceptors (авторизация)

```typescript
// lib/api/client.ts
const client = axios.create({
  baseURL: process.env.NEXT_PUBLIC_API_URL,
});

// Request: добавить токен
client.interceptors.request.use((config) => {
  const token = useAuthStore.getState().accessToken;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// Response: обработка 401 + refresh
client.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401 && !error.config._retry) {
      error.config._retry = true;
      try {
        await useAuthStore.getState().refreshTokens();
        return client(error.config);
      } catch {
        useAuthStore.getState().logout();
        window.location.href = '/login';
      }
    }
    return Promise.reject(error);
  }
);
```

**API:** `POST /auth/login`

**UX-детали:**
- Поле пароля: иконка `Eye`/`EyeOff` для показа/скрытия.
- При отправке: кнопка показывает спиннер, поля блокируются.
- Ошибка `INVALID_CREDENTIALS`: инлайн-сообщение под формой, поля трясутся (`shake` animation, 300ms).
- Ошибка `ACCOUNT_DISABLED`: красный баннер «Аккаунт деактивирован. Обратитесь к другому администратору.»
- При успехе: сохранить `access_token`, `refresh_token`, данные `admin` в Zustand + `httpOnly cookie` через API Route Next.js. Редирект на `/`.
- `Enter` отправляет форму.

**Валидация (Zod):**
```typescript
const schema = z.object({
  email: z.string().email('Введите корректный email'),
  password: z.string().min(1, 'Введите пароль'),
});
```

### 4.2 Выход из системы

- Кнопка «Выйти» в нижней части sidebar.
- При клике: `POST /auth/logout` с текущим refresh_token, затем очистить store, редирект на `/login`.
- Не показывать модалку подтверждения — это лишнее.

**Прогресс-бар:**
- Цвет: `--accent` при < 80%, `--warning` при 80–99%, `--status-completed` при 100%.
- Ширина = `(collected_amount / goal_amount) * 100%`.
- Если `is_permanent = true` или `goal_amount = null`: отображать «Бессрочный» без прогресс-бара.

**Бейджи статусов кампании:**

| Статус | Символ | Цвет |
|---|---|---|
| `active` | ● | green |
| `draft` | ◑ | gray |
| `paused` | ⏸ | orange |
| `completed` | ✓ | blue |
| `archived` | 🗃 | dark gray |

**Контекстное меню `[···]` для строки:**
- Открыть
- Редактировать
- Опубликовать _(если draft)_
- Приостановить _(если active)_
- Возобновить _(если paused)_
- Досрочно закрыть _(если active)_
- Записать офлайн-платёж

### 7.4 Управление статусами кампании

#### Публикация (`draft`/`paused` → `active`)

Кнопка «Опубликовать». Без дополнительного подтверждения (неразрушительное действие). Тост: «Кампания опубликована».

#### Приостановка (`active` → `paused`)

Кнопка «Приостановить». Без подтверждения. Тост: «Кампания приостановлена».

#### Завершение (`active` → `completed`)

Кнопка «Завершить». **Модалка подтверждения:**
```
Завершить кампанию?
━━━━━━━━━━━━━━━━━

Кампания «Сбор на лечение Маши» будет завершена.

Автоматически:
• Все привязанные подписки (89) будут перераспределены
• Все доноры (234) получат push-уведомление

        [Отмена]  [Завершить]
```

#### Досрочное закрытие (`active` → `completed` + `closed_early=true`)

Кнопка «Досрочно закрыть». **Модалка с обязательным текстом:**
```
Досрочное закрытие
━━━━━━━━━━━━━━━━━

Введите сообщение для доноров. Оно будет показано
в карточке кампании и в push-уведомлении.

Комментарий для пользователей *
┌────────────────────────────────────┐
│ Нам удалось собрать 45 000 ₽ из    │
│ 100 000 ₽. Все средства переданы   │
│ фонду.                             │
└────────────────────────────────────┘
min 10 символов, max 500

        [Отмена]  [Закрыть кампанию]
```

Кнопка «Закрыть кампанию» — красная.

#### Архивация (`completed` → `archived`)

Простое подтверждение: «Кампания будет архивирована. Это действие нельзя отменить.»

#### Force Realloc

Кнопка «Принудительная реаллокация» (доступна всегда, видна только в меню `[···]`).

Модалка:
```
Принудительная реаллокация?
━━━━━━━━━━━━━━━━━

Все подписки кампании «Сбор на лечение Маши»
будут немедленно переключены на другие кампании.

⚠️ Пользователи получат уведомления.

        [Отмена]  [Переключить подписки]
```

После успеха: тост «Переключено N подписок».

## 15. Состояния, ошибки и edge-cases

### 15.1 Состояния загрузки

| Тип | Реализация |
|---|---|
| Первая загрузка страницы | Skeleton UI (shimmer) |
| Загрузка после действия (кнопка) | Кнопка показывает спиннер + disabled |
| Background refresh | Незаметно, без блокировки UI |
| Бесконечный список | Кнопка «Загрузить ещё» (не авто-scroll) |

### 15.2 Коды ошибок API → UI-сообщения

| Код ошибки | Сообщение в UI |
|---|---|
| `INVALID_CREDENTIALS` | Неверный email или пароль |
| `ACCOUNT_DISABLED` | Аккаунт деактивирован |
| `TOKEN_EXPIRED` | Сессия истекла. Войдите снова |
| `NOT_FOUND` | Запись не найдена |
| `INN_ALREADY_EXISTS` | Фонд с таким ИНН уже зарегистрирован |
| `INVALID_STATUS_TRANSITION` | Недопустимый переход статуса |
| `DUPLICATE_OFFLINE_PAYMENT` | Похожий платёж уже зафиксирован |
| `FILE_TOO_LARGE` | Файл слишком большой |
| `INVALID_FILE_FORMAT` | Недопустимый формат файла |
| `ACHIEVEMENT_CODE_EXISTS` | Код достижения уже используется |
| `ADMIN_EMAIL_EXISTS` | Email уже занят |
| Сеть недоступна | Нет соединения. Проверьте интернет |
| 500 | Внутренняя ошибка сервера. Попробуйте позже |

### 15.3 Оптимистичные обновления

Для простых toggle-операций (деактивировать/активировать, `is_active`) — применять оптимистичное обновление через TanStack Query:
1. Сразу обновить UI.
2. Если запрос ошибся — откатить.

### 15.4 Навигация с несохранёнными изменениями

При inline-редактировании: если пользователь пытается уйти со страницы с несохранёнными данными — показать нативный браузерный `beforeunload` диалог или кастомный.

## 16. Авторизация и управление токенами

### 16.1 Хранение токенов

- `access_token`: в памяти (`Zustand store`) — не в `localStorage` (защита от XSS).
- `refresh_token`: в `httpOnly cookie` через Next.js API Route (защита от XSS).

### 16.2 Поток обновления токена

1. Axios interceptor ловит `401`.
2. Вызывает `POST /auth/refresh` с `refresh_token` из cookie.
3. Обновляет `access_token` в Zustand.
4. Повторяет оригинальный запрос.
5. Если refresh тоже вернул `401` — редирект на `/login`.

### 16.3 Защита маршрутов

В `app/(admin)/layout.tsx`:
```typescript
// Проверить наличие токена в Zustand.
// Если нет — redirect('/login').
// TanStack Query инициализировать с токеном.
```

### 16.4 Персистентность сессии

При перезагрузке страницы:
1. На клиенте Zustand store пустой (нет access_token).
2. Layout вызывает `POST /auth/refresh` (cookie с refresh_token автоматически отправляется).
3. Получить новый access_token → сохранить в Zustand.
4. Продолжить работу.

# Анализ ТЗ и пробелы текущей админки

## Что описывает [docs/porubly_admin_frontend_tz.md](docs/porubly_admin_frontend_tz.md)

По смыслу документ задаёт:

1. **Продукт и UX** — плотный «операторский» интерфейс (таблицы, фильтры, контекстные меню, подтверждения деструктивных действий), тёмная дизайн-система (CSS-переменные, типографика Inter + Geist Mono, сетка 4px, радиусы, анимации 150ms).
2. **Layout** — фиксированный sidebar 240px, **topbar 52px** с **breadcrumbs + «Обновить» + опционально `[...]**`.
3. **Стек** (ориентир) — Next App Router, TanStack Query/Table, Zustand, RHF+Zod, Axios, Sonner, Recharts, собственный UI-kit.
4. **По разделам** — отдельные макеты и сценарии для: логина, дашборда (**в т.ч. блок «Топ кампании»**), фондов (аватар в таблице, **меню `···**`), кампаний (превью в списке, **меню `···**` на строке, модалки статусов, вкладки деталки с полным набором полей/форм), пользователей (**меню `···**`, подтверждения), админов (**двойной ввод пароля**, «(Вы)», стиль деактивированных), выплат (**вкладки «Баланс» / «История»**, кнопка «Выплатить» с pre-fill), достижений (**toggle активности**, меню), логов (расширенные фильтры, **бейджи причин**, **раскрытие строки** у уведомлений), медиа (**drag-and-drop + прогресс + состояния**).
5. **§14–§16** — детали таблиц (sticky header, skeleton), модалки, маппинг кодов ошибок, **оптимистичные обновления** для toggle, **beforeunload** при несохранённых правках, схема токенов (у вас это уже через BFF).

Навигация и основные маршруты в целом **совпадают** с тем, что есть в [src/app/(admin)](src/app/(admin)).

## Сценарии кампаний: API вызывается, но без требуемых по ТЗ подтверждений

В [campaigns/[id]/page.tsx](src/app/(admin)/campaigns/[id]/page.tsx) сейчас без модалок:

- **Завершить** (`POST .../complete`) — в ТЗ §7.4 обязательная модалка с последствиями (подписки, push).
- **Архивировать** (`POST .../archive`) — в ТЗ простое подтверждение.
- **Force Realloc** — в ТЗ отдельная модалка с предупреждением; сейчас прямой вызов.

Досрочное закрытие и офлайн-платёж уже ближе к ТЗ (модалки есть).

## UX списков и деталей (ТЗ vs реализация)

- **Контекстное меню `···**`: в ТЗ для фондов, кампаний, пользователей, достижений; в коде в основном клик по ссылке в таблице, без меню действий.
- **Дашборд §5.4**: блок **«Топ кампании»** (таблица). В `**GET /stats/overview**` из [api_admin.md](docs/api_admin.md) **нет** поля со списком кампаний — это **пробел ТЗ/продукта относительно API**, а не только фронта. Варианты: договорить отдельный эндпоинт/поле, или собрать «топ» клиентски из `GET /campaigns` (ограничение по смыслу и нагрузке).
- **Списки фондов/кампаний**: в ТЗ колонки превью/аватар, **debounce поиска 300ms** (частично перекрыто `useDeferredValue`), визуальные статусы с точками/символами как в таблице ТЗ.
- **Пользователи**: поиск «email, имя, **телефон**» — зависит от того, ищет ли бэкенд по телефону; **бейджи donor/patron**, меню действий из списка, **ConfirmDialog** для grant-patron / deactivate ([users/[id]/page.tsx](src/app/(admin)/users/[id]/page.tsx) сейчас без этих модалок).
- **Администраторы**: создание без **подтверждения пароля** и без **eye** на обоих полях; в списке нет пометки **«(Вы)»** и **opacity** для неактивных; смена пароля на деталке — одно поле в карточке, а не отдельная модалка «Сменить пароль».
- **Медиа §13**: компонент [media-upload-button.tsx](src/components/media/media-upload-button.tsx) — кнопка + input; нет DnD, прогресс-бара, клиентских проверок размера/MIME до запроса.

## §14–§16 (общие требования)

- **Таблицы**: нет **sticky** header и **skeleton**-строк при первой загрузке ([DataTable](src/components/ui/data-table.tsx)).
- **Ошибки §15.2**: нет единого слоя маппинга всех кодов из таблицы ТЗ (частично через [errors.ts](src/lib/api/errors.ts) и разрозненные `toast`).
- **Оптимистичные обновления** для activate/deactivate и аналогичных — не внедрены.
- **beforeunload** при несохранённых правках на деталках — нет.
- **Topbar**: нет вторичной кнопки `**[...]**` с контекстными действиями страницы.
- **Типографика**: заголовки страниц не везде в стиле «20px / Geist Mono» как в §1.2.

## Итог

- **Все основные группы Admin API** из сводки в конце ТЗ уже дергаются с фронта, кроме явного **PATCH благодарностей** и **неполного использования query-параметров** логов и **полей статистики кампании**.
- Крупнейшие **именно UX/ТЗ-пробелы при готовом бэкенде**: редактирование благодарностей, фильтры и отображение логов, выплаты по макету с вкладками и «Выплатить», подтверждения для complete/archive/force-realloc, контекстные меню в таблицах, достижения `is_active`, медиа-аплоадер по §13, пользователи/админы — подтверждения и мелкие детали списков.
- **«Топ кампании» на дашборде** — в текущем виде **не закрывается одним только фронтом** без расширения `stats/overview` или отдельного контракта API.
